## 1. 改远程文件要 SSH，WSL 本地改无效 ❌

**场景：** 用户让改博客签名为「赛博炼丹师」、姓名改为银白色。我在 WSL 本机改 `profileConfig.ts` 和 `ProfileCardSimple.tsx`，但用户说没变化。

**原因：** 博客代码在远程服务器 `139.199.5.238` 上（`/home/ubuntu/blog/`），本地 WSL 没有代码副本。改本地文件完全没传到服务器。

**教训：** 
- 对于远端项目，**直接 SSH 改远程文件**，别绕一圈
- 或者本地改完手动 `scp` + 远程 `pnpm build`
- **先确认项目路径在哪再动手**

**正确做法：**
```bash
ssh -i ~/.ssh/server_key ubuntu@139.199.5.238 "sed -i ..."  # 直接远程改
# 或：本地写脚本 → scp 到远程 → ssh 执行
```

## 2. sed 不要用来改 JSX 多行代码 ❌

**场景：** 改 `ProfileCardSimple.tsx` 的 `style={{ }}` 多行属性时，用 `sed` 直接替换，结果把语法搞坏了。

**原因：** `sed` 的行匹配和多行替换能力很差，特别是有单引号、花括号嵌套的 JSX。

**教训：**
- JSX/TSX 多行编辑 → **用 Python 脚本做 find-and-replace**，不要用 `sed`
- 或者用 `patch` 工具（Hermes 内置的 find-and-replace 支持模糊匹配）

**推荐做法：**
```python
# 写临时 Python 脚本，scp 到远程执行
with open(path, 'r') as f:
    content = f.read()
content = content.replace(old_string, new_string, 1)
with open(path, 'w') as f:
    f.write(content)
```

## 3. 直接 SSH 执行 Python heredoc 引号爆炸 💥

**场景：** 试图在 SSH 命令里直接嵌入 Python heredoc，遇到引号嵌套、特殊字符时频繁报错。

**原因：** bash heredoc 内的 Python 字符串包含双引号、反斜杠等，bash 和 Python 两层解析导致转义地狱。

**教训：**
- 不要在 SSH 命令里直接写复杂 Python heredoc
- **写本地 `.py` 文件 → `scp` 到 `/tmp/` → `ssh ... python3 /tmp/xxx.py`**
- 这个模式最干净、最可靠

**推荐做法：**
```bash
# 1. 本地写脚本
write_file /tmp/fix_xxx.py

# 2. scp + 执行
scp -i ~/.ssh/server_key /tmp/fix_xxx.py ubuntu@host:/tmp/
ssh -i ~/.ssh/server_key ubuntu@host "python3 /tmp/fix_xxx.py"
```

## 4. Next.js / Astro build 后 Docker bind mount 会掉 ⚠️

**场景：** 主页 `zzr233.online` 404、博客 403。`dist/` 目录文件完好，但 nginx 容器内看不到。

**原因：** Docker bind mount 跟踪的是 **inode**。当 `pnpm build` 或 `npm run build` 重新生成 `dist/` 或 `out/` 目录时，旧目录被删除、新建目录获得新 inode，挂载点失效。

**教训：**
- 每次 `pnpm build` / `npm run build` 后，**必须 `docker restart nginx-home`**
- 这不是 bug，是 Docker bind mount 的固有行为（overlay 文件系统下）
- 可以在 build 脚本里加钩子自动重启

**正确做法：**
```bash
cd /home/ubuntu/blog && pnpm build && docker restart nginx-home
```

## 5. 导航高亮逻辑：先理解业务再下手 🤔

**场景：** 博客导航栏白色高亮框应该默认在「文章」，我第一次写成了首页高亮「主页」。

**原因：** 我以为「主页」是指博客首页。但用户说「主页」是外链到 `zzr233.online`，博客所有页面都应该高亮「文章」。

**教训：**
- **理解每个菜单项的性质**：是本站页面还是外链？
- 对于外链菜单项，永远不会被高亮
- 博客类网站，**默认高亮「文章」更合理**

**最终逻辑：**
```typescript
const getActiveNavIndex = (pathname: string): number => {
  const path = pathname.replace(/\/$/, '') || '/';
  if (path.startsWith('/friends')) return 2;
  if (path.startsWith('/guestbook')) return 3;
  if (path.startsWith('/gallery') || path.startsWith('/anime') || path.startsWith('/bangumi')) return 4;
  if (path.startsWith('/sponsor') || path.startsWith('/about')) return 5;
  return 1; // 默认高亮「文章」
};
```

## 6. Live2DWidget 删了 import 还在打包 🎭

**场景：** 发现 Live2DWidget 有 650KB JS，从 `MainGridLayout.astro` 删掉了 import 和引用，rebuild 后文件还在 `dist/_astro/` 里。

**原因：** 这个文件只是**没有被任何 HTML 引用**的**孤儿文件**。Astro 在构建过程中可能因为缓存或某些发现机制保留了它。检查 `dist/index.html` 确认没有 `<script src="Live2DWidget...">` 引用。

**教训：**
- **孤立的 bundle 文件不一定会被加载**，检查 HTML 确认是否有引用
- 确认 `grep -l 'Live2DWidget' dist/*.html` 无结果即可放心
- 文件占磁盘空间但不影响加载速度
- Astro build 清理：`rm -rf dist .astro && pnpm build`

## 7. 壁纸闪烁：最简单方案往往最好 🎯

**场景：** 用户说第一次进博客背景会「卡一下」，从半截图跳到全屏透明。我分析 SSR/CSR 不一致，写了内联脚本来修复闪烁。

**用户一句话：** 「改成默认全屏透明不就好了」

**教训：**
- **先问自己有没有更简单的方案**
- 默认 `mode: "banner"` → 改为 `mode: "overlay"`，一行配置解决
- 不需要内联脚本、不需要复杂逻辑
- 有时候最直接的修改就是最好的修复

## 总结：Hermes 运维黄金法则

| 法则 | 说明 |
|------|------|
| 🎯 **先问简单方案** | 别一上来就写复杂脚本，先想想改一行配置能不能解决 |
| 🔗 **确认项目在哪** | 远程项目要 SSH 操作，本地改没用 |
| 🐍 **复杂文件改动用 Python** | JSX/多行替换不用 sed，用 Python find-and-replace |
| 📦 **build 后重启容器** | Docker bind mount 会掉，build 完记得 `docker restart` |
| 🏗️ **scp + ssh 模式** | 复杂脚本本地写好 → scp 到远程 → 执行 |
| 🧠 **理解业务再动手** | 导航高亮哪个菜单、背景什么模式，先理清需求 |
| 🔍 **检查孤儿文件** | bundle 有文件不代表 HTML 引用它，先 grep 确认 |
